## 03. Operating System and Process

### Learning Goal

운영체제가 하드웨어 자원을 프로그램에 배분하는 방식을 이해하고, 프로세스·스레드·파일 시스템·권한을 서버 운영의 언어로 설명한다.

### 1. 운영체제의 역할

운영체제(OS)는 애플리케이션과 하드웨어 사이의 관리자다.

| 영역 | 운영체제가 하는 일 |
|---|---|
| 프로세스 | 실행, 중단, 스케줄링, 프로세스 간 격리 |
| 메모리 | 가상 주소 제공, RAM 할당·회수, 보호 |
| 파일 시스템 | 파일·디렉터리·권한·메타데이터 관리 |
| 장치 | 디스크, GPU, 네트워크 카드용 드라이버 제공 |
| 사용자와 권한 | 계정, 그룹, 읽기·쓰기·실행 권한 관리 |
| 네트워크 | 소켓과 TCP/IP 통신 기능 제공 |

서버 환경에서 Linux가 널리 쓰이는 이유는 원격 관리, 자동화, 패키지 관리, 컨테이너 및 클라우드 생태계와 잘 결합되기 때문이다.

### 2. 프로그램, 프로세스, 스레드

- **프로그램**: 저장장치에 있는 실행 코드
- **프로세스**: 프로그램이 실행되어 메모리와 자원을 배정받은 상태
- **스레드**: 프로세스 안에서 실제 명령을 수행하는 실행 흐름

프로세스는 기본적으로 서로 다른 메모리 공간을 사용한다. 같은 프로세스의 스레드는 코드와 힙 같은 자원을 공유하므로 통신은 쉽지만, 잘못된 공유는 경쟁 상태를 만들 수 있다.

### 3. 스케줄링과 컨텍스트 스위칭

실행 가능한 작업이 CPU 코어보다 많으면 OS 스케줄러가 CPU 시간을 나누어 준다. CPU가 한 작업의 레지스터와 실행 위치를 저장하고 다른 작업 상태를 불러오는 것을 컨텍스트 스위칭이라고 한다.

동시 작업 수를 무작정 늘리면 다음 비용이 생긴다.

- 컨텍스트 스위칭 증가
- CPU 캐시 적중률 저하
- 메모리 사용량 증가
- 공유 자원 잠금 경쟁
- 대기열과 지연 시간 증가

따라서 서버의 워커나 스레드 수는 CPU 코어, 작업 특성, 외부 I/O 대기 시간을 기준으로 조정해야 한다.

### 4. 동시성, 병렬성, 비동기

| 개념 | 의미 | 서버 예시 |
|---|---|---|
| 동시성 | 여러 작업의 진행을 함께 관리 | 여러 사용자 요청을 번갈아 처리 |
| 병렬성 | 여러 작업을 물리적으로 동시에 실행 | 여러 CPU 코어에서 계산 수행 |
| 비동기 | 기다리는 동안 실행 흐름을 점유하지 않음 | DB·외부 API 응답 대기 중 다른 요청 처리 |

비동기는 자동으로 더 빠르다는 뜻이 아니다. I/O 대기 시간을 효율적으로 활용하는 구조이며, CPU 계산 자체를 병렬로 만들어 주지는 않는다.

### 5. 가상 메모리

각 프로세스는 자신만의 연속된 메모리 공간을 가진 것처럼 보지만, OS는 가상 주소를 실제 RAM 페이지에 매핑한다. 이 구조는 프로세스 격리, 메모리 보호, 유연한 할당을 가능하게 한다.

RAM 압박이 커지면 OS는 페이지를 스왑으로 내보낼 수 있다. 스왑 사용 증가는 단순한 저장공간 소비가 아니라 메모리 병목 신호다.

### 6. 파일 시스템과 데이터 형식

파일 시스템은 경로, 디렉터리, 소유자, 권한, 생성·수정 시간 등을 관리한다.

| 개념 | 의미 |
|---|---|
| 절대 경로 | 루트부터 시작하는 전체 위치 |
| 상대 경로 | 현재 작업 위치를 기준으로 한 위치 |
| 마운트 | 별도 저장장치를 디렉터리 트리에 연결 |
| 권한 | 사용자·그룹·기타 사용자별 읽기/쓰기/실행 허용 |
| 로그 회전 | 로그가 디스크를 가득 채우지 않도록 분할·보관·삭제 |

데이터는 정형(SQL 테이블), 반정형(JSON·로그), 비정형(이미지·음성·PDF)으로 나눌 수 있다. API는 JSON을 많이 사용하고 분석은 표 구조를 선호하며 AI 입력은 비정형 데이터 비중이 높다.

### 7. 환경변수와 비밀값

설정은 코드와 분리해야 한다. 실행 환경마다 다른 DB 주소, 포트, 기능 플래그는 환경변수나 설정 시스템으로 제공한다. API 키, 토큰, DB 비밀번호는 Git에 커밋하지 않고 Secret Manager 등 별도 비밀 관리 체계에 보관한다.

최소 권한 원칙에 따라 프로세스는 업무 수행에 필요한 파일, 네트워크, 데이터베이스 권한만 가져야 한다. 서버 프로세스를 항상 관리자 권한으로 실행하는 것은 위험하다.

### Technical Literacy Check

- 프로그램, 프로세스, 스레드를 구분할 수 있는가?
- 동시성·병렬성·비동기의 차이를 설명할 수 있는가?
- 컨텍스트 스위칭이 지나치면 왜 성능이 떨어지는가?
- 파일 권한과 비밀값 관리가 운영 보안에 왜 중요한가?

### What I learned

운영체제는 여러 프로그램이 CPU, 메모리, 파일, 장치를 안전하게 공유하도록 중재한다. 서버의 동시 처리 능력은 스레드 수를 늘리는 문제가 아니라 스케줄링, I/O 대기, 메모리와 공유 자원을 함께 설계하는 문제다.

### Questions I can now ask

- 이 서비스는 몇 개의 프로세스와 워커로 실행되는가?
- 작업이 CPU 계산 중심인가, I/O 대기 중심인가?
- 스왑, 파일 디스크립터, 디스크 공간에 운영 한계가 있는가?
- 설정과 비밀값은 코드 및 이미지와 분리되어 있는가?

## 04. Server and Web Communication

### Learning Goal

서버를 물리 장비가 아니라 요청에 응답하는 역할로 이해하고, 클라이언트 요청이 AI 서비스 내부를 통과하는 흐름을 설명한다.

### 1. 클라이언트와 서버

클라이언트는 요청을 보내는 프로그램이고 서버는 요청을 받아 정해진 기능을 수행한 뒤 응답하는 프로그램이다. 노트북도 서버 프로그램을 실행하면 서버가 될 수 있다. 실제 운영에서는 안정성, 보안, 확장성을 위해 데이터센터나 클라우드의 장비를 사용한다.

```text
Client -> Request -> Server
Client <- Response <- Server
```

브라우저는 화면을 보여주는 동시에 HTTP 클라이언트 역할을 한다. API 서버는 인증과 비즈니스 로직을 담당하고, 데이터베이스·캐시·모델 서버 같은 다른 서버의 클라이언트가 되기도 한다.

### 2. 서버 환경의 구분

| 환경 | 목적 | 주요 특징 |
|---|---|---|
| Local | 개인 개발과 빠른 실험 | `localhost`, 외부 공개 아님 |
| Development | 팀 통합 테스트 | 변경이 잦고 테스트 데이터 사용 |
| Staging | 운영 직전 검증 | 운영과 유사한 구성 |
| Production | 실제 사용자 요청 처리 | 안정성, 보안, 관측, 복구 필수 |

환경마다 설정과 데이터가 분리되어야 한다. 개발용 인증 우회나 테스트 키가 운영에 섞이면 보안 사고로 이어질 수 있다.

### 3. IP, 포트, 소켓

- **IP 주소**는 네트워크에서 장치를 찾는 주소다.
- **포트**는 한 장치 안에서 요청을 받을 프로그램을 구분한다.
- **소켓**은 IP, 포트, 전송 프로토콜을 이용해 통신하는 프로그램 인터페이스다.

| 포트 | 대표 용도 |
|---:|---|
| 22 | SSH |
| 80 | HTTP |
| 443 | HTTPS |
| 5432 | PostgreSQL |
| 3306 | MySQL |

포트 번호는 관례일 뿐 보안을 보장하지 않는다. 데이터베이스 포트는 일반 인터넷에 노출하기보다 내부 네트워크와 방화벽으로 접근 주체를 제한해야 한다.

### 4. 일반적인 요청 처리 흐름

1. 클라이언트가 요청을 전송한다.
2. 로드밸런서 또는 게이트웨이가 요청을 수신한다.
3. API 서버가 입력 형식과 인증 정보를 검증한다.
4. 인가 정책으로 사용자의 행동 권한을 확인한다.
5. 캐시나 데이터베이스에서 필요한 데이터를 읽는다.
6. 비즈니스 로직 또는 모델 서버 호출을 수행한다.
7. 결과를 JSON 등의 형식으로 직렬화한다.
8. 상태 코드와 함께 응답하고 로그·메트릭을 기록한다.

사용자가 보는 한 번의 요청이 내부에서는 여러 네트워크 호출로 확장된다. 내부 의존성 하나가 느리거나 실패해도 전체 요청에 영향을 준다.

### 5. 로드밸런서와 확장

로드밸런서는 요청을 여러 서버 인스턴스에 분산하고, 상태 점검에 실패한 인스턴스를 요청 대상에서 제외한다.

| 방식 | 의미 | 장단점 |
|---|---|---|
| Scale Up | 한 서버의 CPU·RAM·GPU 강화 | 단순하지만 장비 한계와 단일 장애 위험 |
| Scale Out | 서버 인스턴스 수 증가 | 탄력적이지만 분산 상태 관리가 필요 |

API 서버는 상태를 외부 DB·캐시로 분리하면 수평 확장이 쉬워진다. GPU 모델 서버는 장비 비용, 모델 로딩 시간, 배치 효율 때문에 더 신중한 확장 정책이 필요하다.

### 6. 타임아웃, 재시도, 멱등성

네트워크 호출은 실패할 수 있다. 서버는 무한정 기다리지 않도록 타임아웃을 두고, 일시적 실패에는 제한적으로 재시도한다. 그러나 재시도는 부하를 늘려 장애를 악화할 수 있으므로 지수 백오프와 최대 횟수가 필요하다.

**멱등성**은 같은 요청을 여러 번 실행해도 결과 상태가 같다는 성질이다. 조회 요청이나 멱등 키를 적용한 결제 요청은 네트워크 재시도 때 중복 처리를 줄일 수 있다.

### 7. AI 서비스에서의 특수한 고려

- 모델 추론은 일반 API보다 오래 걸릴 수 있어 별도 타임아웃 전략이 필요하다.
- 응답 전체를 기다리지 않고 토큰을 스트리밍하면 첫 응답 체감 시간을 줄일 수 있다.
- 입력 크기와 동시 요청 수가 GPU 메모리 및 처리량에 직접 영향을 준다.
- 추론 요청 실패 시 무조건 재시도하면 비용과 대기열이 급증할 수 있다.

### Technical Literacy Check

- 서버가 장비가 아니라 역할이라는 점을 설명할 수 있는가?
- 로컬, 개발, 스테이징, 운영 환경을 구분할 수 있는가?
- IP와 포트의 역할을 구분할 수 있는가?
- Scale Up과 Scale Out의 선택 기준을 말할 수 있는가?

### What I learned

한 번의 사용자 요청은 단일 서버 안에서 끝나지 않는다. 요청은 여러 서버와 저장소를 순차적으로 거치며, 각 구간의 타임아웃·재시도·확장 전략이 전체 서비스의 안정성과 비용을 결정한다.

### Questions I can now ask

- 요청은 어느 서버와 내부 의존성을 어떤 순서로 거치는가?
- 각 호출의 타임아웃과 재시도 정책은 무엇인가?
- 운영 환경이 개발·스테이징과 데이터 및 권한 수준에서 분리되어 있는가?
- 확장 시 상태, 세션, GPU 모델 로딩은 어떻게 처리하는가?

## 05. API, HTTP, DNS, TLS

### Learning Goal

사용자가 도메인을 입력한 순간부터 API 응답을 받기까지 주소 확인, 연결, 암호화, 요청·응답 규칙이 어떻게 이어지는지 이해한다.

### 1. DNS: 이름을 주소로 변환하기

도메인은 사람이 기억하기 쉬운 이름이고 IP 주소는 네트워크가 목적지를 찾는 데 사용하는 주소다. DNS는 도메인에 대응하는 레코드를 조회한다.

단순화한 흐름은 다음과 같다.

1. 브라우저와 운영체제가 로컬 캐시를 확인한다.
2. 재귀 DNS 리졸버에 도메인을 질의한다.
3. 필요하면 루트, TLD, 권한 있는 네임서버를 따라간다.
4. A/AAAA 등의 레코드와 TTL을 응답받아 캐시한다.
5. 클라이언트가 얻은 IP 주소로 연결을 시도한다.

DNS가 잘못되면 애플리케이션 서버가 정상이어도 사용자는 접속할 수 없다. 레코드 변경은 TTL과 각 계층의 캐시 때문에 모든 사용자에게 즉시 반영되지 않을 수 있다.

### 2. TCP와 전송 계층

전통적인 HTTP/1.1과 HTTP/2는 일반적으로 TCP 위에서 동작한다. TCP는 연결, 순서 보장, 손실 재전송, 흐름 제어를 제공한다. HTTP/3는 QUIC를 사용하며 UDP 위에서 전송 안정성과 암호화를 구현한다.

UDP는 연결 설정과 순서 보장을 기본 제공하지 않아 단순하고 지연에 민감한 통신에 활용된다. “UDP는 빠르고 TCP는 느리다”로 단정하기보다 애플리케이션이 요구하는 신뢰성·혼잡 제어·연결 특성을 봐야 한다.

### 3. TLS와 HTTPS

HTTPS는 HTTP를 TLS로 보호한다. TLS는 다음을 제공한다.

- **기밀성**: 중간에서 내용을 읽기 어렵게 암호화
- **무결성**: 전송 중 변조 여부 확인
- **서버 인증**: 인증서와 도메인을 통해 접속한 서버의 신원 확인

TLS 연결 과정에서는 지원 버전과 암호 스위트를 협상하고, 서버 인증서를 검증하며, 세션 키를 만든다. 인증서 만료, 도메인 불일치, 신뢰 체인 오류가 있으면 브라우저가 연결을 차단할 수 있다.

### 4. HTTP 요청과 응답

HTTP 요청은 메서드, 대상 URL, 헤더, 선택적 본문으로 구성된다.

```http
POST /api/chat HTTP/1.1
Host: example.com
Authorization: Bearer <token>
Content-Type: application/json

{"message":"이번 달 매출을 요약해줘"}
```

응답은 상태 코드, 헤더, 선택적 본문으로 구성된다.

```http
HTTP/1.1 200 OK
Content-Type: application/json

{"answer":"전월 대비 12% 증가했습니다."}
```

#### 주요 메서드

| 메서드 | 일반적 의미 |
|---|---|
| GET | 자원 조회 |
| POST | 처리 요청 또는 자원 생성 |
| PUT | 자원 전체 교체 |
| PATCH | 자원 일부 수정 |
| DELETE | 자원 삭제 |

#### 주요 상태 코드

| 코드 | 의미 | 확인할 것 |
|---:|---|---|
| 200/201 | 성공/생성 성공 | 응답 스키마와 실제 상태 |
| 400 | 잘못된 요청 | 입력 형식과 검증 메시지 |
| 401 | 인증 필요 또는 실패 | 토큰 유무·만료 |
| 403 | 권한 없음 | 인가 정책 |
| 404 | 자원을 찾을 수 없음 | URL·자원 존재 여부 |
| 429 | 요청 제한 초과 | rate limit과 재시도 시점 |
| 500 | 서버 내부 오류 | 서버 로그와 request ID |
| 503 | 일시적 서비스 불가 | 과부하·점검·의존성 상태 |

### 5. API는 시스템 간 계약이다

API 계약에는 URL과 메서드뿐 아니라 요청·응답 필드, 자료형, 필수 여부, 오류 형식, 인증 방식, 버전, 제한량이 포함된다. 계약이 명확해야 클라이언트와 서버가 독립적으로 변경될 수 있다.

REST는 자원을 URL로 표현하고 HTTP 의미를 활용하는 대표적 스타일이다. 모든 API가 REST일 필요는 없으며 실시간 양방향 통신에는 WebSocket, 스키마 중심 질의에는 GraphQL, 내부 고성능 통신에는 gRPC 같은 선택지도 있다.

### 6. 인증과 인가

| 개념 | 질문 | 예시 |
|---|---|---|
| 인증(Authentication) | 누구인가? | 로그인, 토큰 검증 |
| 인가(Authorization) | 무엇을 할 수 있는가? | 관리자 기능, 문서 접근 권한 |

API 키, 세션 쿠키, 액세스 토큰, OAuth는 상황에 따라 사용된다. 토큰을 URL에 넣거나 로그에 그대로 남기면 유출 위험이 있다. HTTPS는 전송 중 데이터를 보호하지만 잘못된 권한 설계나 서버 내부 유출까지 해결하지는 않는다.

### 7. API Gateway

서비스가 커지면 외부 요청의 공통 정책을 API Gateway에서 처리할 수 있다.

- 사용자·서비스 인증 확인
- rate limit과 quota 적용
- backend·model version으로 routing
- request/response logging과 request ID 부여
- schema·payload size 검증
- 보안 정책과 차단 규칙
- 사용량 측정과 과금 단위 수집
- 장애 backend 우회와 timeout 정책

AI API에서는 token 사용량 제한, 사용자별 quota, 작은 모델과 큰 모델의 routing, 입력·출력 안전 필터가 중요할 수 있다. Gateway가 모든 보안을 해결하는 것은 아니며 각 backend도 권한과 입력을 검증해야 한다.

### What I learned

웹 요청은 단순히 URL을 열어 서버에 닿는 과정이 아니다. DNS로 주소를 찾고, 전송 연결과 TLS 보안 채널을 만든 뒤, HTTP와 API 계약에 따라 데이터를 교환한다. 어느 계층에서 실패했는지에 따라 해결 방법이 완전히 달라진다.
