## 06. Performance: Latency and Throughput

### Learning Goal

성능을 단일 “속도”로 표현하지 않고 지연 시간, 처리량, 자원 사용률, 오류율, 대기열을 함께 읽는다.

### 1. 지연 시간과 처리량

| 지표 | 의미 | 사용자 관점 |
|---|---|---|
| Latency | 요청 하나가 완료되는 데 걸린 시간 | 얼마나 기다리는가 |
| Throughput | 단위 시간에 완료하는 작업량 | 얼마나 많은 사용자를 처리하는가 |
| Concurrency | 한 시점에 처리 중인 작업 수 | 동시에 몇 요청이 시스템에 있는가 |
| Utilization | 자원이 사용되는 비율 | CPU·GPU가 얼마나 바쁜가 |
| Error rate | 전체 요청 중 실패 비율 | 얼마나 자주 실패하는가 |

처리량을 늘리기 위해 요청을 묶어 GPU 배치로 처리하면 자원 효율은 좋아질 수 있지만, 배치가 모일 때까지 기다리는 개별 요청의 지연 시간은 늘 수 있다. 성능 목표에는 항상 우선순위와 트레이드오프가 있다.

### 2. 평균보다 분포

평균은 극단적으로 느린 요청을 숨길 수 있다.

| 백분위수 | 해석 |
|---|---|
| p50 | 요청 절반이 이 시간 이하에 완료 |
| p95 | 요청 95%가 이 시간 이하에 완료 |
| p99 | 요청 99%가 이 시간 이하에 완료 |

예를 들어 p50이 1초이고 p99가 20초라면 대부분은 빠르지만 일부 사용자의 경험은 매우 나쁘다. 지역, 입력 크기, 사용자 등급, 모델 종류별로 분포를 나누어 봐야 원인을 찾을 수 있다.

AI 스트리밍 서비스에서는 다음 두 지표도 유용하다.

- **TTFT(Time to First Token)**: 첫 토큰이 보이기까지의 시간
- **Inter-token latency**: 이후 토큰 사이의 간격 또는 생성 속도

전체 응답 시간이 같아도 첫 토큰이 빨리 도착하면 체감은 달라진다.

평균이 1초여도 p99가 15초라면 일부 사용자는 매우 나쁜 경험을 한다. 이를 tail latency 문제라고 한다. 평균을 낮추는 최적화와 느린 꼬리 요청을 줄이는 최적화는 다를 수 있다.

### 3. 대기열과 포화

도착률이 처리율에 가까워질수록 작은 변동에도 요청이 대기한다. 처리 능력을 넘는 상태가 지속되면 대기열이 계속 커지고, 타임아웃과 재시도가 추가 트래픽을 만들어 장애를 증폭시킨다.

```text
Arrival rate < Service rate  -> queue remains manageable
Arrival rate >= Service rate -> queue grows -> latency -> timeout -> retry
```

따라서 자원 사용률을 항상 100%에 가깝게 유지하는 것이 효율적이라는 생각은 위험하다. 급격한 트래픽과 느린 요청을 흡수할 여유가 필요하다.

### 4. 병목 찾기

1. 사용자 관점의 증상과 시간 범위를 정한다.
2. 요청량, 오류율, p95/p99 지연을 확인한다.
3. API, DB, 캐시, 모델 서버 등 구간별 시간을 트레이스로 나눈다.
4. 같은 시간대의 CPU, RAM, GPU, VRAM, 디스크, 네트워크를 비교한다.
5. 최근 배포, 설정 변경, 외부 서비스 장애를 확인한다.
6. 가설을 하나씩 실험하거나 재현한다.

상관관계는 원인이 아니다. CPU 사용률이 높아진 시점에 지연이 늘었다고 해서 즉시 CPU가 근본 원인이라고 단정할 수 없다. DB 지연으로 대기 작업이 쌓이며 CPU 부담까지 늘었을 수 있다.

### 5. 성능 개선 수단

| 수단 | 기대 효과 | 주의점 |
|---|---|---|
| 캐시 | 반복 계산·조회 감소 | 오래된 데이터, 무효화 |
| 배치 | GPU 및 I/O 효율 향상 | 개별 요청 대기 증가 |
| 비동기 처리 | 긴 작업을 요청 경로에서 분리 | 상태 추적과 재처리 필요 |
| Scale Out | 처리 용량 증가 | 비용과 분산 상태 복잡도 |
| 데이터 축소 | 전송·연산량 감소 | 품질 저하 가능성 |
| 인덱스·쿼리 개선 | DB 지연 감소 | 쓰기 비용과 저장공간 증가 |

#### AI 추론 최적화

| 기법 | 의미 | 기대 효과 | 주의점 |
|---|---|---|---|
| Quantization | weight·activation 정밀도 축소 | VRAM·비용 감소 | 품질과 하드웨어 지원 검증 |
| Distillation | 큰 모델의 출력을 작은 모델 학습에 활용 | 작은 모델 확보 | 원 모델 능력 완전 복제 불가 |
| Batching | 여러 요청을 한 번에 계산 | GPU 처리량 증가 | 개별 대기 시간 증가 |
| Caching | 반복 입력 또는 중간 결과 재사용 | 계산·API 비용 감소 | 최신성·개인정보·키 설계 |
| Model routing | 난이도·정책에 따라 모델 선택 | 품질 대비 비용 최적화 | 잘못된 routing과 복잡성 |
| Streaming | 결과를 조금씩 전송 | TTFT와 체감 지연 개선 | 총 계산 시간은 줄지 않을 수 있음 |

챗봇은 latency와 TTFT가 중요하고, 야간 문서 분류는 개별 latency보다 throughput과 마감 시간이 중요할 수 있다. “빠르게”의 의미를 업무별로 정의해야 한다.

최적화 전에 목표를 수치로 정해야 한다. “더 빠르게”보다 “정상 입력의 p95를 3초 이하로, 오류율을 0.1% 이하로”가 실행 가능한 목표다.

### 6. 신뢰성과 비용

성능을 높이기 위해 항상 서버를 늘리면 비용이 증가한다. 반대로 자원을 지나치게 줄이면 지연과 오류가 늘어난다. 서비스 수준 목표(SLO)는 사용자에게 중요한 가용성과 지연 기준을 정하고, 자원과 개발 우선순위를 연결하는 도구다.

AI 시스템에서는 GPU 유휴 비용과 대기 시간 사이의 균형이 특히 중요하다. 오토스케일링도 모델 로딩 시간과 GPU 확보 지연을 고려해야 한다.

### What I learned

빠른 시스템은 평균값 하나로 설명되지 않는다. 요청 분포, 처리량, 오류율, 대기열, 자원 사용률을 같은 시간축에서 보고 사용자 경험과 비용 사이의 목표를 정해야 한다.

### Questions I can now ask

- 처리량을 높이는 변경이 개별 요청 지연과 비용에 어떤 영향을 주는가?

## 07. Cache, Queue, Batch, Stream

### Learning Goal

데이터의 위치와 작업 시점을 조절하는 네 가지 패턴을 이해하고, 속도·최신성·신뢰성·복잡성의 트레이드오프를 설명한다.

### 1. 캐시: 가까운 곳에서 재사용하기

캐시는 생성하거나 가져오는 비용이 큰 데이터를 더 빠른 위치에 임시 보관한다.

| 위치 | 예시 |
|---|---|
| CPU 캐시 | 최근 접근한 메모리 데이터 |
| 브라우저·CDN | 이미지, CSS, JavaScript |
| 애플리케이션 캐시 | 자주 조회하는 API 결과 |
| DB 캐시 | 데이터 페이지와 쿼리 결과 |
| AI 서비스 캐시 | 동일 입력 결과, 임베딩, 세션 정보 |

캐시의 핵심 질문은 “무엇을 저장할까?”보다 **언제 오래된 것으로 판단하고 어떻게 갱신할까?**다.

- TTL: 일정 시간이 지나면 만료
- Cache-aside: 앱이 캐시를 먼저 확인하고 없으면 원본에서 읽어 저장
- Write-through: 쓰기를 캐시와 원본에 함께 반영
- 무효화: 원본 변경 시 관련 캐시 삭제 또는 갱신

캐시 적중률이 높아도 잘못된 데이터가 오래 남으면 품질 문제가 된다. 개인정보나 사용자별 권한이 있는 응답은 캐시 키와 접근 범위를 특히 주의해야 한다.

### 2. 메시지 큐: 생산자와 소비자 분리하기

메시지 큐는 요청을 받은 서비스와 실제 작업을 수행하는 워커를 분리한다.

```text
Producer -> Queue -> Consumer/Worker -> Result
```

적합한 작업은 이메일 발송, 파일 변환, 리포트 생성, 데이터 전처리, 알림, 로그 처리처럼 즉시 완료하지 않아도 되는 일이다.

#### 운영 시 핵심 개념

- **Acknowledgement**: 소비자가 작업 성공을 큐에 알림
- **Retry**: 일시적 실패를 다시 처리
- **Dead-letter queue**: 반복 실패 메시지를 별도로 격리
- **Visibility timeout**: 처리 중인 메시지가 즉시 다른 워커에 보이지 않도록 함
- **Backpressure**: 소비 속도보다 생산 속도가 빠를 때 유입을 조절

대부분의 큐는 중복 전달 가능성을 고려해야 한다. 소비자는 같은 메시지를 두 번 받아도 안전하도록 멱등성을 갖추는 것이 중요하다.

### 3. 배치 처리

배치는 데이터를 일정 기간 또는 일정량 모아 한꺼번에 처리한다.

- 일일 매출 집계
- 정기 백업
- 대량 데이터 변환
- 모델 학습용 데이터 생성
- GPU 추론 요청 묶음 처리

한 번의 초기화와 I/O로 많은 항목을 처리해 효율적이지만, 결과의 최신성이 낮아진다. 배치 실패 시 어느 지점부터 재실행할지, 부분 성공을 어떻게 기록할지 설계해야 한다.

### 4. 스트림 처리

스트림 처리는 지속적으로 들어오는 이벤트를 실시간 또는 준실시간으로 처리한다.

- 이상 거래 탐지
- 실시간 알림
- 클릭 이벤트 집계
- 서비스 상태 모니터링
- 모델 입력·출력 이벤트 분석

이벤트는 늦게 도착하거나 순서가 바뀌거나 중복될 수 있다. 이벤트 시간과 처리 시간, 윈도우, 중복 제거, 체크포인트가 중요한 이유다.

### 5. 네 패턴 비교

| 패턴 | 바꾸는 것 | 장점 | 주요 위험 |
|---|---|---|---|
| Cache | 데이터의 위치 | 낮은 지연과 원본 부하 감소 | 오래된 값, 무효화 |
| Queue | 작업의 결합도와 실행 시점 | 피크 흡수, 장애 격리 | 지연, 중복, 적체 |
| Batch | 처리 단위와 시점 | 높은 효율, 낮은 단가 | 낮은 최신성, 재실행 |
| Stream | 연속 이벤트의 처리 방식 | 빠른 반응 | 순서·중복·운영 복잡성 |

하나가 다른 하나를 완전히 대체하지 않는다. 예를 들어 실시간 이벤트를 스트림으로 수집하고, 큐로 작업을 분배하며, 캐시로 최근 집계를 제공하고, 야간 배치로 정산 결과를 확정할 수 있다.

### 6. AI 서비스 사례

문서 분석 기능을 예로 들면 다음과 같이 구성할 수 있다.

1. 업로드 API가 파일을 저장하고 작업 ID를 반환한다.
2. 분석 메시지를 큐에 넣는다.
3. 워커가 파일을 전처리하고 모델 추론을 수행한다.
4. 결과를 DB에 저장하고 상태 이벤트를 발행한다.
5. 자주 조회하는 완료 결과는 캐시한다.
6. 운영 분석을 위해 하루치 사용량을 배치 집계한다.

이 구조는 긴 분석을 HTTP 연결 안에서 계속 기다리는 방식보다 피크 트래픽과 실패에 강하지만, 상태 조회, 재처리, 중복 방지 설계가 추가된다.

### Technical Literacy Check

- 캐시 적중과 캐시 무효화를 설명할 수 있는가?
- 메시지 큐가 순간 트래픽을 흡수하는 원리를 설명할 수 있는가?
- 배치와 스트림의 최신성·복잡성 차이를 말할 수 있는가?
- 중복 메시지와 반복 실패 작업을 어떻게 다룰지 질문할 수 있는가?

### What I learned

캐시, 큐, 배치, 스트림은 단순한 제품 이름이 아니라 데이터 위치와 작업 시간을 재배치하는 설계 패턴이다. 응답 속도와 처리 효율을 높이는 대신 최신성, 중복, 재처리, 운영 복잡성을 책임져야 한다.

### Questions I can now ask

- 캐시의 원본, 키, TTL, 무효화 조건은 무엇인가?
- 큐 적체량과 가장 오래된 메시지의 대기 시간을 관찰하는가?
- 소비자는 중복 전달에도 안전한가?
- 이 요구사항은 정말 실시간이어야 하는가, 배치로 충분한가?

## 08. Container, Cloud, Observability

### Learning Goal

애플리케이션을 일관되게 배포하고, 변화하는 트래픽에 맞춰 운영하며, 장애를 관찰·복구하는 현대 인프라의 기본 구조를 이해한다.

### 1. 컨테이너와 Docker

컨테이너는 애플리케이션과 필요한 라이브러리·설정을 격리된 실행 환경으로 묶는다. 가상머신이 운영체제 전체를 가상화하는 것과 달리 컨테이너는 호스트 커널을 공유하므로 일반적으로 가볍고 시작이 빠르다.

| 개념 | 의미 |
|---|---|
| Dockerfile | 이미지를 만드는 절차 |
| Image | 실행 파일과 의존성을 담은 불변 템플릿 |
| Container | 이미지에서 시작한 실행 인스턴스 |
| Registry | 이미지를 저장·배포하는 저장소 |
| Volume | 컨테이너 생명주기와 분리된 데이터 저장소 |

컨테이너는 “내 컴퓨터에서는 된다”는 환경 차이를 줄이지만 이미지를 자동으로 안전하게 만들지는 않는다. 버전 고정, 취약점 점검, 최소 이미지, 비루트 사용자, 비밀값 분리가 필요하다.

AI 컨테이너는 Python·프레임워크뿐 아니라 GPU driver와 호환되는 CUDA runtime, 시스템 library, model artifact에도 민감하다. host driver는 보통 컨테이너와 별도로 관리되므로 이미지에 CUDA를 넣었다는 사실만으로 GPU 실행이 보장되지는 않는다.

### 2. Kubernetes와 오케스트레이션

컨테이너가 많아지면 배치, 재시작, 네트워크, 설정, 확장을 자동 관리할 체계가 필요하다. Kubernetes는 원하는 상태를 선언하면 실제 상태가 그에 가까워지도록 지속적으로 조정한다.

| 기능 | 의미 |
|---|---|
| Scheduling | 자원이 있는 노드에 워크로드 배치 |
| Self-healing | 실패 컨테이너 재시작·교체 |
| Service discovery | 서비스 이름으로 인스턴스 탐색 |
| Rolling update | 가용성을 유지하며 버전 교체 |
| Autoscaling | 부하 지표에 따라 인스턴스 수 조정 |
| Config/Secret | 설정과 비밀값 주입 |

Kubernetes를 쓴다고 자동으로 고가용성이 생기는 것은 아니다. 잘못된 상태 점검, 과도한 자원 제한, 단일 DB 의존성은 그대로 장애 요인이 된다.

### 3. 클라우드의 기본 자원

| 범주 | 대표 기능 |
|---|---|
| Compute | VM, 컨테이너, 서버리스, CPU/GPU 인스턴스 |
| Storage | 블록, 파일, 객체 저장소 |
| Database | 관계형·키값·문서형 관리형 DB |
| Network | VPC, 서브넷, 방화벽, 로드밸런서, CDN |
| IAM | 사용자·서비스 계정과 권한 관리 |
| Operations | 로그, 메트릭, 알림, 감사 기록 |

클라우드는 서버를 보이지 않게 만드는 것이 아니라 API로 빠르게 빌리고 관리하게 한다. 탄력성과 관리 편의성을 얻는 대신 지속 비용, 공급자 의존성, 데이터 이동 비용, 권한 복잡성을 관리해야 한다.

### 4. 관측성의 세 기둥

| 신호 | 답하는 질문 | 예시 |
|---|---|---|
| Logs | 어떤 사건이 발생했는가? | 오류 메시지, request ID, 배포 이벤트 |
| Metrics | 시스템 상태가 시간에 따라 어떻게 변하는가? | 요청률, 오류율, p95, GPU 사용률 |
| Traces | 한 요청이 어느 경로에서 시간을 썼는가? | API -> DB -> model server 구간별 span |

세 신호는 경쟁하지 않는다. 메트릭으로 이상을 감지하고, 트레이스로 느린 구간을 찾고, 해당 구간의 로그로 구체적 원인을 확인할 수 있다.

로그에는 비밀번호, 토큰, 개인정보, 전체 프롬프트를 무분별하게 남기지 않는다. request ID와 trace ID를 사용하면 민감한 원문 없이도 여러 서비스의 기록을 연결할 수 있다.

### 5. 신뢰성, 장애, 복구

서버와 네트워크는 언젠가 실패한다. 현실적인 목표는 실패를 빠르게 감지하고 영향 범위를 줄이며 복구하는 것이다.

- 여러 인스턴스와 로드밸런서로 단일 장애점 축소
- 상태 점검과 자동 재시작
- 데이터 백업과 실제 복구 훈련
- 타임아웃, 제한된 재시도, 서킷 브레이커
- 배포 자동화와 빠른 롤백
- 운영 절차(runbook)와 사고 회고

백업 파일이 존재하는 것과 복구할 수 있는 것은 다르다. 복구 시간 목표(RTO)와 허용 가능한 데이터 손실 범위(RPO)를 정하고 검증해야 한다.

### 6. 보안과 최소 권한

- 외부에는 필요한 포트와 엔드포인트만 공개한다.
- IAM 사용자와 서비스 계정에 필요한 권한만 부여한다.
- 비밀값은 전용 시스템에 저장하고 주기적으로 교체한다.
- 이미지와 의존성 취약점을 점검한다.
- 저장·전송 데이터를 암호화하고 접근 기록을 남긴다.
- GitHub에 키, 토큰, 환자·고객 데이터를 커밋하지 않는다.

특히 임상·의료 관련 데이터는 조직의 법률·보안 기준과 데이터 처리 정책을 먼저 확인해야 한다.

### 7. 비용은 아키텍처의 제약 조건이다

GPU 실행 시간, 유휴 인스턴스, 저장 용량, 로그 수집량, DB 요청, 네트워크 송신은 모두 비용이다. 비용 최적화는 단순 삭감이 아니라 사용자 경험과 신뢰성 목표를 유지하며 낭비를 줄이는 일이다.

태그와 비용 배분, 예산 알림, 자원별 사용률, 모델·기능별 단위 비용을 함께 보면 기술 결정과 사업 영향을 연결할 수 있다.

### What I learned

현대 인프라는 프로그램을 컨테이너로 포장하고 클라우드 자원 위에 배포한 뒤 오케스트레이터로 원하는 상태를 유지한다. 운영 품질은 배포 자체보다 관측, 최소 권한, 장애 복구, 비용 통제까지 연결할 때 만들어진다.
